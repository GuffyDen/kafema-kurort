# PROJECT.md

Внутренняя конституция продукта Kafema. Этот документ должен читать любой разработчик или ИИ-агент перед внесением изменений в проект.

Это не README и не рекламное описание. Здесь зафиксированы продуктовые роли, архитектурные принципы и правила, которые нельзя ломать при развитии Kafema.

## 0. Главная цель продукта

Kafema - это не POS-система.

Kafema - операционная система кофейни.

Продукт должен уменьшать хаос внутри кофейни: ускорять прием заказов, помогать бариста работать без лишних действий, давать собственнику контроль над меню и процессами.

Если есть выбор между количеством функций и скоростью работы сотрудников - всегда выбирать скорость.

Новая функция имеет смысл только тогда, когда она помогает кофейне работать лучше, быстрее или надежнее.

## 1. Что такое Kafema

Kafema - это не просто сайт для заказа кофе. Это операционная экосистема для кофейни, которая связывает клиента, бариста и собственника через одну общую модель меню и заказов.

В системе есть три основных интерфейса:

- Client - клиентское приложение для заказа.
- Bar - рабочее место бариста.
- Manager - панель управления кофейней.

Каждый интерфейс существует для своей роли. Нельзя смешивать их задачи в один общий экран.

### URL-структура интерфейсов

Role и URL - разные понятия.

Интерфейсы продукта:

```text
Client
URL: /

Bar
URL: /bar

Manager
URL: /admin
```

Manager - название бизнес-интерфейса для собственника или управляющего.

Admin - адрес приложения, по которому открывается Manager-интерфейс.

Bar - название рабочей роли и интерфейса бариста, который открывается по адресу `/bar`.

Нельзя переименовывать Manager-сущности в коде только потому, что URL равен `/admin`. Названия `ManagerPanel`, `ManagerStore`, `ManagerLayout`, `ManagerPage`, `ManagerEngine`, `ManagerModel` и аналогичные сущности описывают продуктовую роль, а не URL.

## 2. Главная философия

Каждый интерфейс должен решать задачу своей роли.

Client:

- красиво;
- быстро;
- минимум действий до заказа;
- понятный статус активного заказа.

Bar:

- максимум скорости;
- минимум кликов;
- без лишнего дизайна;
- только рабочая информация;
- действия должны быть очевидны и выполняться одним нажатием.

Manager:

- управление меню;
- категории;
- рабочие зоны;
- библиотека дополнений;
- сотрудники в будущем;
- аналитика в будущем.

Главный принцип: красота важна для клиента, скорость важна для бариста, контроль важен для собственника.

## 3. Роли и интерфейсы

### Client

Клиент:

- открывает меню по QR;
- выбирает товары;
- выбирает варианты и дополнения;
- оформляет заказ;
- видит активный заказ и статус;
- получает понятный номер заказа для выдачи.

Client должен ощущаться как мобильное приложение, а не как лендинг или обычный сайт.

### Bar

Бариста:

- видит заказы;
- работает по колонкам: Новые / В работе / Готовы;
- работает строго в FIFO-порядке;
- использует одну кнопку действия на карточке;
- видит таймеры по стадиям;
- использует отдельную вкладку Стоп-лист;
- управляет стоп-листом по товарам и по вариантам дополнений.

Bar не должен показывать лишнюю аналитику, финансовые данные или настройки меню, которые мешают работе.

### Manager

Собственник или управляющий:

- не работает со стоп-листом ежедневно;
- создает и редактирует пункты меню;
- создает категории;
- создает рабочие зоны;
- создает библиотеку дополнений;
- управляет активностью товаров и доступностью;
- в будущем управляет сотрудниками, ролями, аналитикой и настройками точки.

Manager должен быть понятным инструментом управления кофейней, а не технической простыней форм.

## 4. Menu Engine

Главная сущность меню - Menu Item, а не Product.

Menu Item может быть:

- напиток;
- еда;
- десерт;
- комбо;
- сезонное предложение;
- сертификат;
- любой будущий пункт меню.

Menu Item содержит:

- название;
- описание;
- фото;
- категорию;
- рабочую зону;
- активность;
- наличие;
- порядок отображения;
- варианты;
- подключенные группы дополнений.

Menu Engine должен быть общим источником истины для Client, Bar и Manager. Нельзя создавать отдельную модель товара только для клиента или отдельную модель меню только для бариста.

## 4.1. Single Source of Truth

Каждая бизнес-сущность существует только в одном месте.

Примеры:

- цена овсяного молока хранится только в библиотеке дополнений;
- рабочая зона хранится только у Menu Item;
- категория хранится только у Menu Item;
- цена модификатора не копируется в каждый напиток;
- доступность варианта дополнения хранится у варианта дополнения;
- нельзя копировать данные между сущностями;
- если данные меняются, они должны меняться только в одном месте.

Client, Bar и Manager могут показывать одни и те же данные по-разному, но источник этих данных должен быть один.

Если нужно изменить структуру данных, сначала нужно понять, где находится источник истины. Нельзя решать проблему быстрым копированием поля в соседнюю сущность.

## 4.2. Архитектурная схема

Базовая схема Menu Engine:

```text
Category
    │
    ▼
Menu Item
    │
    ├── Variant
    │
    └── Modifier Group
            │
            └── Modifier Option
```

Пояснения:

- Category нужна клиенту, чтобы меню было понятно сгруппировано.
- Working Zone нужна сотрудникам, чтобы понимать, где готовится позиция.
- Working Zone не показана как родитель в схеме, потому что она не группирует меню для клиента, а связывается с Menu Item как операционное свойство.
- Modifier Group переиспользуется между разными Menu Item.
- Modifier Option принадлежит группе дополнений, а не конкретному товару.
- Menu Item подключает существующие Modifier Group, но не владеет их ценами и вариантами.

Пример:

- Category: Кофе.
- Menu Item: Капучино.
- Working Zone: Бар.
- Variant: 250 мл / 350 мл / 450 мл.
- Modifier Group: Молоко.
- Modifier Option: Овсяное +80.

Цена овсяного молока меняется в Modifier Option и автоматически применяется везде, где используется группа Молоко.

## 5. Категории и рабочие зоны

Категория - для клиента.

Примеры категорий:

- Кофе;
- Выпечка;
- Десерты;
- Холодные напитки;
- Завтраки.

Рабочая зона - для сотрудников.

Примеры рабочих зон:

- Бар;
- Кухня;
- Витрина;
- Холодный цех.

Категория и рабочая зона не одно и то же.

Например:

- Категория: Кофе. Рабочая зона: Бар.
- Категория: Десерты. Рабочая зона: Витрина.
- Категория: Завтраки. Рабочая зона: Кухня.

Бариста видит заказ по рабочим зонам, а не по клиентским категориям. Клиент видит меню по категориям, а не по внутренним зонам приготовления.

## 6. Варианты и дополнения

### Варианты Menu Item

Варианты описывают базовую конфигурацию пункта меню:

- размер;
- вкус;
- объем;
- другой базовый вариант товара.

Примеры:

- Капучино: 250 мл / 350 мл / 450 мл.
- Лимонад: Манго / Клубника / Лимон.
- Сертификат: 1000 / 3000 / 5000.

Вариант может менять цену.

### Группы дополнений

Группы дополнений - отдельные переиспользуемые сущности. Они не должны дублироваться внутри каждого товара.

Примеры групп:

- Молоко;
- Сиропы;
- Сахар;
- Температура;
- Дополнительный шот.

Группа дополнений поддерживает:

- название;
- обязательная/необязательная;
- один выбор / множественный выбор;
- варианты;
- сортировку.

Вариант дополнения содержит:

- название;
- доплату;
- активность;
- наличие;
- порядок.

Важный принцип: если овсяное молоко подорожало, цена меняется в одном месте, а не в каждом кофе отдельно. Если овсяное молоко недоступно, оно выключается в одном месте и пропадает во всех напитках, где подключена группа молока.

## 6.1. Product Principles

Любое изменение должно соответствовать минимум одному принципу:

- ускоряет работу бариста;
- уменьшает вероятность ошибки;
- упрощает обучение новых сотрудников;
- не дублирует данные;
- не нарушает Menu Engine;
- масштабируется на сеть кофеен.

Если изменение не соответствует ни одному принципу - его нельзя добавлять.

Хорошая задача для Kafema делает систему проще в эксплуатации, даже если внутри требует аккуратной архитектурной работы.

## 6.2. Что запрещено

Запрещено:

- создавать отдельные модели товаров для Client, Bar и Manager;
- дублировать Modifier Group внутри Menu Item;
- хранить цену модификатора внутри товара;
- создавать бизнес-логику внутри UI-компонентов;
- смешивать обязанности ролей;
- усложнять работу бариста ради красивого интерфейса;
- добавлять всплывающие окна там, где бариста должен работать быстро;
- превращать Manager в рабочее место бариста;
- превращать Bar в аналитический кабинет;
- добавлять новые поля без понимания, где находится источник истины.

Если кажется, что проще сделать временный дубль данных, нужно остановиться и пересмотреть модель.

## 7. Стоп-лист

Стоп-лист - ежедневный рабочий инструмент бариста.

Он должен быть в `/bar`, а не главным разделом Manager.

Стоп-лист должен:

- иметь поиск;
- позволять быстро найти товар или вариант дополнения;
- переключать наличие одним нажатием;
- работать без всплывающих окон;
- не спрашивать подтверждение;
- не добавлять лишние шаги.

Стоп-лист товаров:

- если товар недоступен, клиент не может его заказать;
- в текущем клиентском меню недоступный товар должен исчезать из выдачи или быть явно недоступен, в зависимости от утвержденного клиентского UX.

Стоп-лист дополнений:

- если недоступно овсяное молоко, оно исчезает во всех напитках, где оно подключено;
- сами напитки остаются доступными;
- другие варианты той же группы продолжают работать.

Manager может видеть и редактировать наличие в карточке пункта меню, но ежедневная работа с быстрым стоп-листом принадлежит Bar.

## 8. Bar UX

Рабочее место бариста - это операционная доска, а не красивый сайт.

Правила:

- три колонки: Новые / В работе / Готовы;
- FIFO: старые сверху, новые снизу;
- при переходе в следующую колонку заказ добавляется вниз;
- никаких сложных анимаций;
- одна кнопка действия;
- в Новые и В работе главным является состав заказа;
- в Готовы главным является номер заказа и клиент;
- номер заказа использовать через `#`, не `№`;
- слово "Заказ" в карточке не использовать;
- цена и сумма в `/bar` не показываются;
- телефон нужен для связи с клиентом, но второстепенный;
- состав заказа группируется по рабочим зонам;
- дизайн должен помогать скорости, а не украшать.

Кнопки по стадиям:

- Новые: Принять;
- В работе: Готов;
- Готовы: Выдан.

Таймеры:

Новые:

- до 1 минуты - серый;
- 1-2 минуты - желтый;
- больше 2 минут - красный.

В работе:

- до 5 минут - серый;
- 5-7 минут - желтый;
- больше 7 минут - красный.

Готовы:

- до 3 минут - серый;
- 3-10 минут - желтый;
- больше 10 минут - красный.

После выдачи заказа не нужно спрашивать подтверждение. Бариста должен работать быстро.

## 9. Manager UX

Manager не должен быть одной длинной технической формой.

Каждый экран решает одну задачу.

Разделы:

- Меню;
- Библиотека дополнений;
- Категории;
- Рабочие зоны;
- Стоп-лист не основной ежедневный раздел;
- Аналитика позже;
- Настройки позже.

Экран Меню отвечает на вопрос: "Какие пункты меню у меня есть?"

Экран Дополнения отвечает на вопрос: "Какие группы дополнений существуют?"

Экран Категории отвечает на вопрос: "Как клиент видит меню?"

Экран Рабочие зоны отвечает на вопрос: "Кто/где готовит этот пункт?"

Manager должен управлять структурой бизнеса, а не повторять рабочее место бариста.

## 9.1. MVP First

Порядок разработки:

1. Рабочая логика.
2. Архитектура.
3. UX.
4. Дизайн.
5. Анимации.

Никогда не менять этот порядок.

Сначала система должна правильно работать. Потом она должна быть правильно устроена. Потом ей должно быть удобно пользоваться. Только после этого можно заниматься визуальной полировкой и анимациями.

Если дизайн мешает скорости бариста или усложняет понимание процесса, дизайн нужно упростить.

## 10. Технические принципы

Сейчас для MVP можно использовать `localStorage` и общий client-side store.

Но архитектура должна быть готова к переходу на Supabase.

Разделение ответственности:

- Vercel хранит и запускает сайт, но не хранит бизнес-данные.
- GitHub хранит код.
- Supabase в будущем хранит данные и realtime-события.

Supabase в будущем будет хранить:

- заказы;
- меню;
- пользователей;
- роли;
- фото;
- realtime-статусы;
- stop-list.

Фото:

- в MVP можно временно хранить `data URL` / `base64`;
- в продукте фото должны храниться в Supabase Storage или аналоге;
- владелец не должен вставлять URL вручную;
- должен быть нормальный Upload фото.

Общие правила разработки:

- не ломать архитектуру Menu Engine;
- не плодить параллельные модели товара;
- не смешивать Client, Bar и Manager;
- интерактивные части Next.js должны быть Client Components;
- Server Components использовать там, где нет browser-only API;
- состояние MVP может быть локальным, но границы store должны позволять заменить его на Supabase;
- после изменений запускать `npm run build`.

## 12. Инструкция для будущих задач

Перед любой задачей разработчик или ИИ должен:

1. Прочитать `PROJECT.md`.
2. Не нарушать принципы ролей.
3. Не смешивать Client, Bar и Manager в один интерфейс.
4. Не добавлять функционал, который усложняет работу бариста.
5. Сохранять архитектуру Menu Engine.
6. После изменений запускать `npm run build`.

Если новая задача противоречит этому документу, сначала нужно явно обсудить изменение принципов продукта, а не молча менять архитектуру.
