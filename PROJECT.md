# PROJECT.md

Внутренняя конституция продукта Kafema. Этот документ должен читать любой разработчик или ИИ-агент перед внесением изменений в проект.

Это не README и не рекламное описание. Здесь зафиксированы продуктовые роли, архитектурные принципы и правила, которые нельзя ломать при развитии Kafema.

## 0. Главная цель продукта

Kafema - это не POS-система.

Kafema - операционная система кофейни.

Kafema больше не является самостоятельной системой учета кафе.

Операционный источник истины для меню, цен, категорий, модификаторов, стоп-листов и наличия - iiko.

Kafema строит клиентскую витрину, онлайн-оплату, очередь бариста и управленческий слой поверх данных, синхронизированных из iiko.

Продукт должен уменьшать хаос внутри кофейни: ускорять прием заказов, помогать бариста работать без лишних действий, давать собственнику контроль над меню и процессами.

Если есть выбор между количеством функций и скоростью работы сотрудников - всегда выбирать скорость.

Новая функция имеет смысл только тогда, когда она помогает кофейне работать лучше, быстрее или надежнее.

## 1. Что такое Kafema

Kafema - это не просто сайт для заказа кофе. Это операционная экосистема для кофейни, которая связывает клиента, бариста и собственника через одну общую модель витрины и заказов, построенную поверх iiko.

В системе есть три основных интерфейса:

- Client - клиентское приложение для заказа.
- Bar - рабочее место бариста.
- Manager - панель управления кофейней.

Каждый интерфейс существует для своей роли. Нельзя смешивать их задачи в один общий экран.

### URL-структура интерфейсов

Role и URL - разные понятия.

Интерфейсы продукта:

```text
Client
URL: /

Bar
URL: /bar

Manager
URL: /admin
```

Manager - название бизнес-интерфейса для собственника или управляющего.

Admin - адрес приложения, по которому открывается Manager-интерфейс.

Bar - название рабочей роли и интерфейса бариста, который открывается по адресу `/bar`.

Нельзя переименовывать Manager-сущности в коде только потому, что URL равен `/admin`. Названия `ManagerPanel`, `ManagerStore`, `ManagerLayout`, `ManagerPage`, `ManagerEngine`, `ManagerModel` и аналогичные сущности описывают продуктовую роль, а не URL.

## 2. Главная философия

Каждый интерфейс должен решать задачу своей роли.

Client:

- красиво;
- быстро;
- минимум действий до заказа;
- понятный статус активного заказа.

Bar:

- максимум скорости;
- минимум кликов;
- без лишнего дизайна;
- только рабочая информация;
- действия должны быть очевидны и выполняться одним нажатием.

Manager:

- управление витриной поверх iiko-товаров;
- фото, красивые названия и описания;
- видимость на сайте;
- сортировка, бейджи и рекомендации;
- настройки QR-витрины;
- сотрудники в будущем;
- аналитика в будущем.

Главный принцип: красота важна для клиента, скорость важна для бариста, контроль важен для собственника.

## 3. Роли и интерфейсы

### Client

Клиент:

- открывает меню по QR;
- выбирает товары, варианты и дополнения, полученные из iiko;
- оформляет заказ через paid-first flow;
- видит активный заказ и статус;
- получает понятный номер заказа для выдачи.

Paid-first flow для клиента:

1. Клиент собирает корзину.
2. Система создает платеж в ЮKassa.
3. ЮKassa подтверждает оплату через webhook `payment.succeeded`.
4. Только после успешного webhook создается оплаченный заказ.
5. Оплаченный заказ появляется на экране бариста.
6. Заказ отправляется в iiko как оплаченный.

Client должен ощущаться как мобильное приложение, а не как лендинг или обычный сайт.

### Bar

Бариста:

- видит оплаченные онлайн-заказы;
- работает по колонкам: Новые / В работе / Готовы;
- работает строго в FIFO-порядке;
- использует одну кнопку действия на карточке;
- видит таймеры по стадиям;
- использует отдельную вкладку Стоп-лист;
- видит статусы синхронизации с iiko;
- в будущем видит не только онлайн-заказы, но и заказы с кассы / iikoFront.

Каждый заказ должен иметь источник:

- `ONLINE` - заказ из клиентской QR-витрины;
- `CASHIER` - заказ с кассы / iikoFront;
- `DELIVERY` - заказ из доставки или внешнего канала.

Bar не должен показывать лишнюю аналитику, финансовые данные или настройки меню, которые мешают работе.

### Manager

Собственник или управляющий:

- не создает товары с нуля внутри Kafema;
- редактирует витринный слой поверх товаров, синхронизированных из iiko;
- управляет фото, красивым названием, описанием, видимостью, сортировкой, бейджами и рекомендациями;
- настраивает QR-витрину;
- видит наличие и стоп-лист как данные из iiko;
- в будущем управляет сотрудниками, ролями, аналитикой и настройками точки.

Manager должен быть понятным инструментом управления кофейней, а не технической простыней форм.

## 4. Menu Engine

Главная сущность клиентского меню - Menu Item, а не Product.

Menu Item в Kafema - это не самостоятельный товарный справочник. Это витринное представление позиции, синхронизированной из iiko.

iiko является source of truth для:

- товарной номенклатуры;
- базовых названий;
- цен;
- категорий;
- вариантов / размеров;
- модификаторов;
- стоп-листов;
- наличия.

Menu Item может быть:

- напиток;
- еда;
- десерт;
- комбо;
- сезонное предложение;
- сертификат;
- любой будущий пункт меню.

Menu Item содержит:

- внешний идентификатор iiko;
- данные iiko: цена, категория, варианты, модификаторы, наличие;
- витринный слой Kafema: фото, красивое название, описание, видимость на сайте, сортировка, бейджи, рекомендации, настройки отображения.

Menu Engine должен быть общей моделью чтения для Client, Bar и Manager. Но source of truth для операционных данных меню - iiko.

Kafema может хранить только витринные данные, которые не являются учетными данными кафе:

- фото;
- красивое название;
- описание;
- видимость на сайте;
- сортировку;
- бейджи;
- рекомендации;
- настройки QR-витрины.

Нельзя создавать отдельную модель товара только для клиента или отдельную модель меню только для бариста.

## 4.1. Single Source of Truth

Каждая бизнес-сущность существует только в одном месте.

Сейчас главный внешний source of truth - iiko.

Примеры:

- цена овсяного молока хранится в iiko, а не в Kafema;
- цена товара хранится в iiko, а не в карточке витрины;
- категория хранится в iiko и приходит через синхронизацию;
- модификаторы и их цены приходят из iiko;
- стоп-лист и наличие приходят из iiko;
- фото и красивое описание хранятся в Kafema как витринный слой;
- нельзя копировать данные между сущностями;
- если данные меняются, они должны меняться только в одном месте.

Client, Bar и Manager могут показывать одни и те же данные по-разному, но источник операционных данных должен быть один.

Если нужно изменить структуру данных, сначала нужно понять, где находится источник истины. Нельзя решать проблему быстрым копированием поля в соседнюю сущность.

## 4.2. Архитектурная схема

Базовая схема Menu Engine с учетом iiko:

```text
iiko
  │
  ├── Category
  ├── Menu Item
  │       ├── Variant
  │       └── Modifier Group
  │               └── Modifier Option
  ├── Prices
  └── Stop-list / Availability

Kafema
  │
  └── Storefront Overlay
          ├── photo
          ├── displayName
          ├── description
          ├── visibility
          ├── sortOrder
          ├── badges
          ├── recommendations
          └── QR storefront settings
```

Пояснения:

- Category нужна клиенту, чтобы меню было понятно сгруппировано, но источник категории - iiko.
- Working Zone нужна сотрудникам, чтобы понимать, где готовится позиция.
- Modifier Group переиспользуется между разными Menu Item и приходит из iiko.
- Modifier Option принадлежит группе дополнений, а не конкретному товару, и приходит из iiko.
- Kafema Storefront Overlay не меняет цену, состав модификаторов или наличие.
- Manager редактирует Storefront Overlay, а не создает учетный товар с нуля.

Пример:

- Category: Кофе.
- Menu Item: Капучино.
- Working Zone: Бар.
- Variant: 250 мл / 350 мл / 450 мл.
- Modifier Group: Молоко.
- Modifier Option: Овсяное +80.

Цена овсяного молока меняется в iiko и автоматически применяется везде после синхронизации.

## 5. Категории и рабочие зоны

Категория - для клиента.

Источник категорий - iiko. Kafema может управлять только витринным отображением категории, если это не конфликтует с iiko.

Примеры категорий:

- Кофе;
- Выпечка;
- Десерты;
- Холодные напитки;
- Завтраки.

Рабочая зона - для сотрудников.

Примеры рабочих зон:

- Бар;
- Кухня;
- Витрина;
- Холодный цех.

Категория и рабочая зона не одно и то же.

Если рабочие зоны приходят из iiko / iikoFront, Kafema использует их как source of truth. Если для MVP используется локальная настройка рабочих зон, она должна быть изолирована за абстракцией и готова к замене на данные iiko.

Например:

- Категория: Кофе. Рабочая зона: Бар.
- Категория: Десерты. Рабочая зона: Витрина.
- Категория: Завтраки. Рабочая зона: Кухня.

Бариста видит заказ по рабочим зонам, а не по клиентским категориям. Клиент видит меню по категориям, а не по внутренним зонам приготовления.

## 6. Варианты и дополнения

### Варианты Menu Item

Варианты описывают базовую конфигурацию пункта меню и приходят из iiko:

- размер;
- вкус;
- объем;
- другой базовый вариант товара.

Примеры:

- Капучино: 250 мл / 350 мл / 450 мл.
- Лимонад: Манго / Клубника / Лимон.
- Сертификат: 1000 / 3000 / 5000.

Вариант может менять цену. Цена варианта должна приходить из iiko.

### Группы дополнений

Группы дополнений - отдельные переиспользуемые сущности, которые приходят из iiko. Они не должны дублироваться внутри каждого товара.

Примеры групп:

- Молоко;
- Сиропы;
- Сахар;
- Температура;
- Дополнительный шот.

Группа дополнений поддерживает:

- название;
- обязательная/необязательная;
- один выбор / множественный выбор;
- варианты;
- сортировку.

Вариант дополнения содержит:

- название;
- доплату;
- активность;
- наличие;
- порядок.

Важный принцип: если овсяное молоко подорожало, цена меняется в iiko, а не в каждом кофе отдельно. Если овсяное молоко недоступно, оно выключается в iiko и пропадает во всех напитках после синхронизации.

## 6.1. Product Principles

Любое изменение должно соответствовать минимум одному принципу:

- ускоряет работу бариста;
- уменьшает вероятность ошибки;
- упрощает обучение новых сотрудников;
- не дублирует данные;
- не нарушает Menu Engine;
- сохраняет iiko как source of truth для операционных данных;
- сохраняет ЮKassa как source of truth по факту оплаты;
- масштабируется на сеть кофеен.

Если изменение не соответствует ни одному принципу - его нельзя добавлять.

Хорошая задача для Kafema делает систему проще в эксплуатации, даже если внутри требует аккуратной архитектурной работы.

## 6.2. Что запрещено

Запрещено:

- создавать отдельные модели товаров для Client, Bar и Manager;
- дублировать Modifier Group внутри Menu Item;
- хранить цену модификатора внутри товара;
- хранить цены, категории, модификаторы, стоп-лист или наличие как ручную правду в Kafema;
- превращать Manager в редактор создания товаров с нуля;
- создавать оплаченный заказ до подтверждения `payment.succeeded` от ЮKassa;
- хранить секреты, API-ключи, `shopId`, `secretKey` или токены в коде;
- создавать бизнес-логику внутри UI-компонентов;
- смешивать обязанности ролей;
- усложнять работу бариста ради красивого интерфейса;
- добавлять всплывающие окна там, где бариста должен работать быстро;
- превращать Manager в рабочее место бариста;
- превращать Bar в аналитический кабинет;
- добавлять новые поля без понимания, где находится источник истины.

Если кажется, что проще сделать временный дубль данных, нужно остановиться и пересмотреть модель.

## 7. Стоп-лист

Стоп-лист - ежедневный рабочий инструмент бариста, но source of truth для стоп-листа и наличия - iiko.

Он должен быть в `/bar`, а не главным разделом Manager.

Стоп-лист должен:

- иметь поиск;
- позволять быстро найти товар или вариант дополнения;
- переключать наличие одним нажатием;
- работать без всплывающих окон;
- не спрашивать подтверждение;
- не добавлять лишние шаги.

Стоп-лист товаров:

- если товар недоступен, клиент не может его заказать;
- в текущем клиентском меню недоступный товар должен исчезать из выдачи или быть явно недоступен, в зависимости от утвержденного клиентского UX.

Стоп-лист дополнений:

- если недоступно овсяное молоко, оно исчезает во всех напитках, где оно подключено;
- сами напитки остаются доступными;
- другие варианты той же группы продолжают работать.

Manager может видеть наличие в карточке пункта витрины, но не должен становиться главным редактором стоп-листа.

Если Bar меняет стоп-лист, действие должно идти через `MenuProvider` / интеграционный слой iiko. До получения ключей iiko допускается mock/local режим, но интерфейс не должен зависеть от локальной реализации.

## 8. Bar UX

Рабочее место бариста - это операционная доска, а не красивый сайт.

Bar показывает оплаченные онлайн-заказы. В будущем Bar должен также уметь показывать заказы с кассы / iikoFront и доставки.

Правила:

- три колонки: Новые / В работе / Готовы;
- FIFO: старые сверху, новые снизу;
- при переходе в следующую колонку заказ добавляется вниз;
- никаких сложных анимаций;
- одна кнопка действия;
- в Новые и В работе главным является состав заказа;
- в Готовы главным является номер заказа и клиент;
- номер заказа использовать через `#`, не `№`;
- слово "Заказ" в карточке не использовать;
- цена и сумма в `/bar` не показываются;
- телефон нужен для связи с клиентом, но второстепенный;
- состав заказа группируется по рабочим зонам;
- каждый заказ имеет `source`: `ONLINE`, `CASHIER` или `DELIVERY`;
- онлайн-заказ появляется у бариста только после подтвержденной оплаты;
- если отправка в iiko после оплаты временно не удалась, заказ остается видимым бариста;
- статус синхронизации с iiko может быть `IIKO_SYNC_PENDING` или `IIKO_SYNC_FAILED`;
- дизайн должен помогать скорости, а не украшать.

Кнопки по стадиям:

- Новые: Принять;
- В работе: Готов;
- Готовы: Выдан.

Таймеры:

Новые:

- до 1 минуты - серый;
- 1-2 минуты - желтый;
- больше 2 минут - красный.

В работе:

- до 5 минут - серый;
- 5-7 минут - желтый;
- больше 7 минут - красный.

Готовы:

- до 3 минут - серый;
- 3-10 минут - желтый;
- больше 10 минут - красный.

После выдачи заказа не нужно спрашивать подтверждение. Бариста должен работать быстро.

## 9. Manager UX

Manager не должен быть одной длинной технической формой.

Каждый экран решает одну задачу.

Разделы:

- Витрина;
- Товары из iiko;
- Категории из iiko;
- Модификаторы из iiko;
- Настройки QR-витрины;
- Стоп-лист не основной ежедневный раздел;
- Аналитика позже;
- Настройки позже.

Экран Витрина отвечает на вопрос: "Как товары из iiko выглядят для клиента?"

Экран Товары из iiko отвечает на вопрос: "Какие позиции синхронизированы из iiko и какие витринные данные к ним привязаны?"

Экран Модификаторы отвечает на вопрос: "Какие группы дополнений пришли из iiko и как они отображаются клиенту?"

Экран Категории отвечает на вопрос: "Как клиент видит категории, синхронизированные из iiko?"

Экран Настройки QR-витрины отвечает на вопрос: "Как выглядит и ведет себя клиентская витрина конкретной точки?"

Manager должен управлять витринным слоем бизнеса, а не повторять рабочее место бариста и не заменять iiko.

## 9.1. MVP First

Порядок разработки:

1. Рабочая логика.
2. Архитектура.
3. UX.
4. Дизайн.
5. Анимации.

Никогда не менять этот порядок.

Сначала система должна правильно работать. Потом она должна быть правильно устроена. Потом ей должно быть удобно пользоваться. Только после этого можно заниматься визуальной полировкой и анимациями.

Если дизайн мешает скорости бариста или усложняет понимание процесса, дизайн нужно упростить.

## 10. Технические принципы

Сейчас для MVP можно использовать `localStorage`, общий client-side store и mock/local providers.

Но архитектура должна быть готова к подключению iiko, ЮKassa и Supabase.

Разделение ответственности:

- Vercel хранит и запускает сайт, но не хранит бизнес-данные.
- GitHub хранит код.
- iiko хранит меню, цены, категории, модификаторы, стоп-листы и наличие.
- ЮKassa является source of truth по факту оплаты.
- Supabase в будущем хранит витринные данные, заказы, статусы синхронизации, пользователей, роли и realtime-события.

Supabase в будущем будет хранить:

- оплаченные заказы и их статусы;
- витринные данные поверх iiko-товаров;
- пользователей;
- роли;
- фото;
- realtime-статусы;
- статусы отправки в iiko;
- настройки QR-витрины.

### Интеграционные абстракции

Все внешние интеграции должны идти через абстракции:

- `MenuProvider` - получает меню, категории, модификаторы, цены, наличие и стоп-листы из iiko или mock/local режима.
- `OrderProvider` - создает и синхронизирует заказы с iiko или mock/local режимом.
- `PaymentProvider` - создает платежи, проверяет статусы и обрабатывает webhook ЮKassa или mock/local режим.
- `BaristaQueueProvider` - формирует очередь бариста из оплаченных заказов и в будущем объединяет `ONLINE`, `CASHIER`, `DELIVERY`.

UI-компоненты не должны знать, работает ли приложение с iiko, ЮKassa, Supabase или mock/local источником.

До получения ключей iiko и ЮKassa все должно работать в mock/local режиме.

Секреты, API-ключи, `shopId`, `secretKey`, webhook secrets и токены нельзя хранить в коде, коммитить в GitHub или писать в клиентский bundle. Они должны жить только в переменных окружения и серверной инфраструктуре.

### Paid-first orders

Порядок создания онлайн-заказа:

1. Клиент собирает корзину.
2. `PaymentProvider` создает платеж в ЮKassa.
3. Клиент оплачивает заказ.
4. Webhook ЮKassa `payment.succeeded` подтверждает факт оплаты.
5. Только после этого `OrderProvider` создает оплаченный заказ в нашей БД.
6. `BaristaQueueProvider` показывает заказ бариста.
7. `OrderProvider` отправляет заказ в iiko как оплаченный.

Если iiko недоступна после оплаты:

- заказ не теряется;
- заказ остается в нашей БД;
- заказ виден баристе;
- заказ получает статус `IIKO_SYNC_PENDING` или `IIKO_SYNC_FAILED`;
- система повторно отправляет заказ в iiko позже.

Нельзя строить flow, где неоплаченный заказ попадает в Bar как обычный рабочий заказ.

Фото:

- в MVP можно временно хранить `data URL` / `base64`;
- в продукте фото должны храниться в Supabase Storage или аналоге;
- владелец не должен вставлять URL вручную;
- должен быть нормальный Upload фото.

Общие правила разработки:

- не ломать архитектуру Menu Engine;
- не плодить параллельные модели товара;
- не делать Kafema source of truth для iiko-данных;
- все интеграции вести через providers;
- не смешивать Client, Bar и Manager;
- интерактивные части Next.js должны быть Client Components;
- Server Components использовать там, где нет browser-only API;
- состояние MVP может быть локальным, но границы store должны позволять заменить его на Supabase;
- после изменений запускать `npm run build`.

## 12. Инструкция для будущих задач

Перед любой задачей разработчик или ИИ должен:

1. Прочитать `PROJECT.md`.
2. Не нарушать принципы ролей.
3. Не смешивать Client, Bar и Manager в один интерфейс.
4. Не добавлять функционал, который усложняет работу бариста.
5. Сохранять архитектуру Menu Engine как витрины поверх iiko.
6. Не превращать Kafema в source of truth для меню, цен, модификаторов, стоп-листа или наличия.
7. Не обходить `MenuProvider`, `OrderProvider`, `PaymentProvider`, `BaristaQueueProvider`.
8. Не хранить секреты и API-ключи в коде.
9. После изменений запускать `npm run build`.

Если новая задача противоречит этому документу, сначала нужно явно обсудить изменение принципов продукта, а не молча менять архитектуру.
