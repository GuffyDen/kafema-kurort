Согласно локальной документации Next.js 16, страницы и layout в `app/` являются Server Components по умолчанию. Интерактивные части MVP следует выносить в точечные Client Components с директивой `"use client"`, чтобы не раздувать клиентский bundle. Для серверных мутаций можно использовать Server Functions/Server Actions, а для внешних интеграций и webhook/API-сценариев - Route Handlers в `app/api/.../route.ts`.

## Цель MVP

MVP - сервис предзаказа кофе для одной кофейни без маркетплейса и без мульти-точек. Пользователь должен быстро выбрать напитки, настроить опции, указать имя и телефон, выбрать время готовности, отправить заказ и получить номер/статус. Сотрудник кофейни должен видеть входящие заказы и менять их статус.

## Предлагаемый пользовательский сценарий MVP

1. Пользователь открывает главную страницу.
2. Видит статус кофейни: открыта/закрыта, ближайшее доступное время выдачи.
3. Выбирает напитки из меню.
4. Настраивает размер, молоко, сиропы, количество и комментарий.
5. Проверяет корзину.
6. Указывает имя и телефон.
7. Выбирает время самовывоза из доступных слотов.
8. Подтверждает заказ.
9. Видит страницу подтверждения с номером заказа и статусом.

## Границы MVP

Входит:

- меню одной кофейни;
- карточки товаров;
- базовые модификаторы напитков;
- корзина;
- оформление заказа без регистрации;
- выбор времени самовывоза;
- простая админ-панель для заказов;
- статусы заказа: `new`, `accepted`, `preparing`, `ready`, `completed`, `cancelled`;
- хранение заказов и меню в базе данных;
- базовая валидация форм;
- адаптивная верстка для мобильного сценария.

Не входит в MVP:

- онлайн-оплата;
- доставка;
- бонусная программа;
- мульти-кофейни;
- сложная CRM;
- push-уведомления;
- интеграция с кассой;
- личный кабинет клиента;
- полноценная аналитика.

## Архитектура MVP

### Рекомендуемый стек

Базовый стек проекта уже подходит для MVP:

- Next.js App Router как full-stack слой;
- Server Components для страниц, которые читают данные;
- Client Components только для корзины, выбора опций и интерактивных контролов;
- Server Actions для создания заказа и изменения статусов;
- Route Handlers для будущих webhook/integration endpoints;
- TypeScript для доменных типов;
- Tailwind CSS 4 для интерфейса.

Для хранения данных рекомендуется добавить:

- PostgreSQL как основную базу;
- Prisma ORM или Drizzle ORM как типизированный слой доступа к данным.

Для MVP проще выбрать Prisma: она быстрее вводится, понятна команде, удобна для миграций и админских CRUD-сценариев. Если приоритет - максимально легкий runtime и SQL-контроль, можно выбрать Drizzle.

### Доменные сущности

Минимальная модель данных:

- `Product` - напиток или товар: название, описание, цена, активность, категория, сортировка.
- `ProductOptionGroup` - группа опций: размер, молоко, сироп.
- `ProductOption` - конкретная опция: oat milk, large, vanilla syrup, доплата.
- `Order` - заказ: код, имя клиента, телефон, статус, время самовывоза, сумма, комментарий.
- `OrderItem` - позиция заказа: товар, количество, цена на момент покупки.
- `OrderItemOption` - выбранные опции позиции.
- `CafeSettings` - часы работы, шаг слота, минимальное время приготовления, флаг приема заказов.

На MVP меню можно администрировать через seed/миграции или простую внутреннюю страницу. Полноценный редактор меню лучше отложить на следующий этап, если меню редко меняется.

### Маршруты приложения

Публичная часть:

- `/` - меню, корзина, состояние кофейни.
- `/checkout` - оформление заказа.
- `/order/[orderCode]` - подтверждение и статус заказа.

Админская часть:

- `/admin` - обзор текущих заказов.
- `/admin/orders` - список заказов с фильтрами по статусу.

Служебные маршруты:

- `/api/health` - проверка доступности приложения.
- В будущем: `/api/webhooks/...` для оплаты, уведомлений или кассовой интеграции.

### Слои ответственности

- `app/.../page.tsx` - композиция страниц, загрузка данных на сервере.
- `app/_components/...` - UI-компоненты по зонам продукта.
- `app/actions/...` - серверные мутации: создать заказ, сменить статус.
- `app/_lib/db.ts` - клиент базы данных.
- `app/_lib/validation.ts` - схемы валидации входных данных.
- `app/_lib/pickup-slots.ts` - расчет доступных временных слотов.
- `app/_lib/money.ts` - расчет сумм в копейках/центах без float-ошибок.

### Состояние корзины

Для MVP корзину можно держать на клиенте:

- React state в Client Component;
- опционально `localStorage`, чтобы не терять корзину при обновлении страницы;
- финальный расчет цены обязательно повторять на сервере при создании заказа, не доверяя клиентским суммам.

### Безопасность и валидация

- Все Server Actions должны валидировать входные данные.
- Цена заказа рассчитывается на сервере по актуальному меню.
- Админские действия должны быть защищены хотя бы простым паролем/сессией.
- Нельзя хранить секреты в клиентском коде.
- Телефон и имя клиента нужно хранить минимально необходимое время.
- Для админки стоит добавить rate limiting или хотя бы базовую защиту на уровне хостинга.

### Админский интерфейс MVP

Минимально достаточно:

- список заказов за сегодня;
- заметная группировка по статусам;
- карточка заказа: номер, время самовывоза, имя, телефон, позиции, комментарий, сумма;
- кнопки смены статуса;
- ручная отмена заказа;
- автообновление через polling каждые 10-20 секунд.

Realtime через WebSocket/SSE можно отложить, если поток заказов небольшой.

### Дизайн-направление

Так как сервис будет использоваться быстро и часто с телефона, интерфейс должен быть не лендингом, а рабочим ordering flow:

- первый экран сразу показывает меню и возможность заказать;
- крупные зоны касания;
- понятная корзина снизу или отдельной панелью;
- минимальное количество шагов;
- ясное состояние "кофейня принимает заказы" / "заказы закрыты";
- строгая адаптивность под мобильные экраны.

## Риски и решения

- Неверные цены на клиенте: считать финальную сумму только на сервере.
- Дубли заказов: блокировать повторную отправку и использовать идемпотентность при необходимости.
- Заказы вне рабочего времени: централизовать расчет слотов в `pickup-slots.ts`.
- Сложная админка затянет MVP: начать только со статусов заказов, меню менять через seed.
- Слишком большой клиентский bundle: держать страницы серверными, `"use client"` добавлять только для корзины и интерактивных элементов.

## Ближайшее следующее решение

Перед началом реализации нужно выбрать слой хранения данных:

- Prisma + PostgreSQL - рекомендуемый путь для быстрого MVP.
- Drizzle + PostgreSQL - вариант для более явного SQL-контроля.
- Временное хранение в JSON/SQLite - не рекомендуется для реального предзаказа, но возможно для совсем короткого прототипа.
